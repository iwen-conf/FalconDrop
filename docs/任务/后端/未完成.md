## 总目标

基于 `docs/rewrite-analysis/12-confirmed-web-scope.md`，实现 Go 后端：HTTP API、WebSocket、FTP 服务、本地数据卷存储、PostgreSQL 持久化和 Docker Compose 正式部署。

## 技术栈与范围豁免

- 后端主语言：Go。
- Web 服务：Gin 或 Echo 二选一，最终以 spike 和团队熟悉度为准。
- 数据库：PostgreSQL，配置、账号、媒体资产、传输事件和系统元数据均以数据库为真源。
- DB 访问：`pgx` + SQL repository，或 `sqlc` 生成查询层；不使用隐藏 SQL 的重型 ORM 作为首选。
- migration：`goose` 或 `golang-migrate`，必须支持空库初始化和 Compose 环境执行。
- 日志：`slog` 或 `zap`，输出 stdout。
- 认证：本项目按 `12-confirmed-web-scope.md` 豁免注册、多用户和角色；只做唯一默认系统账号登录。Lazycat 创建基线中的注册、双 token、无感刷新默认要求不进入本项目 P0，除非后续产品范围变更。
- 文件存储：只做服务器本地数据卷，不抽象 NAS/S3/MinIO。

## 交付里程碑

### M1 工程与数据基线

- 初始化 Go module、配置、日志、数据库连接、migration、健康检查和 Docker Compose。
- 空库启动后能 seed 唯一系统账号、唯一 FTP 账号和基础 app settings。
- 提供统一错误响应和 request id。

### M2 认证与系统信息

- 默认系统账号可登录、退出和恢复会话。
- `/api/system/info` 返回版本、build hash、系统时间、系统账号、FTP 账号、FTP 状态和存储状态。
- 未登录不能访问受保护 API 和 WebSocket。

### M3 FTP 与本地存储

- Go FTP server 可启动、停止、查询状态。
- 默认 FTP 账号和匿名访问均可用。
- PASV 端口范围、public host 和 Docker 端口映射策略可验证。
- 上传任意格式文件后进入本地数据卷。

### M4 媒体入库与实时事件

- 上传完成后计算 hash、识别 MIME、提取 EXIF、写入 `media_assets` 和 `transfer_events`。
- 同名同 hash 覆盖，同名不同 hash 保留两份。
- 照片 API、资产 API、图片 content API 和删除 API 可用。
- WebSocket 推送上传、覆盖、删除和 FTP 状态变化。

### M5 验收闭环

- 覆盖单元测试、集成测试和 Docker smoke test。
- 主路径可复现：登录 -> 启动 FTP -> 上传照片 -> 入库 -> WebSocket 刷新 -> 删除。

## 任务依赖顺序

1. `01` 和 `02` 是所有后续任务的基础，必须先完成。
2. `03` 依赖 `02` 的账号表和 seed 逻辑。
3. `04` 依赖 `02` 的 `ftp_account`、`app_settings` 和 storage 配置。
4. `05` 依赖 `04` 的上传完成 hook 和 storage backend。

7. `08` 随每个模块落地，不等最后集中补。
