# Web 版已确认范围

本文件记录 Web 重写的最新决策，是 `docs/rewrite-analysis/` 后续实现和拆任务的优先事实源。旧文档中与本文件冲突的 P1/P2 建议均以本文件为准。

## 产品边界

| 决策项 | 已确认结论 |
|---|---|
| 部署形态 | Docker Compose，作为正式部署方式 |
| 平台形态 | 只做 Web，不做 Android、Windows 桌面端和 Tauri 原生能力 |
| 页面范围 | 仅两个主页面：照片、系统信息 |
| 前端语言 | 页面展示文案除专业术语外一律使用中文 |
| 照片页面 | 展示上传的所有照片，支持按 EXIF 时间分类查看 |
| 系统信息页面 | 展示系统版本、系统 hash、系统时间、系统账号、FTP 账号等系统层面信息 |
| Web 账号 | 只需要一个默认账号，不做多用户和角色区分 |
| FTP 账号 | 只需要一个默认账号 |
| FTP 匿名访问 | 支持匿名，不限制 IP |
| FTPS | 不需要 |
| AI 修图 | 不做 |
| 实时通信 | 使用 WebSocket，不使用 SSE |
| 数据库 | PostgreSQL |
| 文件存储 | 只做服务器本地存储，不做 NAS/S3/MinIO 抽象 |
| 上传格式 | 允许所有格式上传 |
| 图库范围 | 照片页面只展示可识别为照片的上传文件；非照片文件可保留上传记录但不进入照片视图 |
| 删除语义 | 直接删除数据库记录和本地文件 |

## 文件规则

| 场景 | 规则 |
|---|---|
| 文件名 | 保留原文件名 |
| 去重依据 | 文件内容 hash |
| 同名且 hash 一致 | 允许覆盖，保持同一内容记录 |
| 同名但 hash 不一致 | 不覆盖；必须保留两份内容，存储层需要生成不冲突的内部路径 |
| 排序时间 | 以 EXIF 拍摄时间为准 |
| 无 EXIF 时间 | 实现时需要提供兜底时间字段，但照片分组优先使用 EXIF 时间 |

## 架构约束

- 后端使用 Go 提供 HTTP API、WebSocket 和 FTP 服务。
- 前端使用 React Web，只保留照片和系统信息两个主要视图。
- 前端页面文案除专业术语外一律使用中文；`Docker Compose`、`PostgreSQL`、`WebSocket`、`FTP`、`EXIF`、`hash`、`API` 等技术名词可保留英文。
- PostgreSQL 是配置、账号、媒体资产、上传事件和系统元数据的唯一持久化数据库。
- Docker Compose 必须包含 Web/API 服务、PostgreSQL、数据卷和 FTP 控制端口/PASV 端口映射。
- 浏览器访问媒体文件必须通过后端受控接口，不能暴露服务器真实路径。
- 系统不做 RBAC、用户管理、审计日志、AI 任务、对象存储、FTPS 证书管理和外部分享链接。

## 建议的核心数据

| 表 | 用途 |
|---|---|
| `system_accounts` | 保存唯一 Web 默认账号及密码 hash |
| `ftp_account` | 保存唯一 FTP 默认账号及密码 hash，匿名策略也可在此或 settings 中保存 |
| `app_settings` | 保存 FTP 端口、PASV 端口、本地存储路径等配置 |
| `media_assets` | 保存上传文件资产、hash、原文件名、内部存储路径、MIME、大小、EXIF 时间和照片识别状态 |
| `transfer_events` | 保存上传、覆盖、删除等传输事件 |

## 首期验收标准

1. 使用 Docker Compose 可以启动 Web/API、PostgreSQL 和本地存储卷。
2. 默认 Web 账号可以登录系统。
3. 系统信息页能展示系统版本、系统 hash、系统时间、系统账号和 FTP 账号。
4. FTP 默认账号和匿名模式均可按配置连接，且不做 IP 限制。
5. FTP 客户端上传任意格式文件后，文件进入本地存储并写入 PostgreSQL。
6. 上传照片文件后，照片页能通过 WebSocket 实时刷新并按 EXIF 时间分类展示。
7. 同名同 hash 上传按覆盖处理；同名不同 hash 不覆盖已有文件。
8. 删除照片会直接删除本地文件和数据库记录。
9. 前端页面除专业术语外均使用中文文案。
